## 한 줄 요약

이 영상은 지난 4년 동안 AI 활용 방식이 **프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → 바이브 코딩 → 하네스 엔지니어링**으로 어떻게 이동해왔는지 정리하면서, 결국 개발자의 경쟁력이 **모델 사용법 자체보다 AI가 일하는 환경과 절차를 설계하는 능력**으로 옮겨가고 있다고 설명한다.

---

## 핵심 메시지

영상의 중심 메시지는 꽤 명확하다.

1. 초반에는 좋은 결과를 얻기 위해 **프롬프트 문구 자체**를 잘 쓰는 것이 중요했다.
2. 이후에는 더 많은 정보와 도구를 연결하는 **컨텍스트 설계**가 중요해졌다.
3. 그리고 누구나 AI로 빠르게 결과물을 뽑는 **바이브 코딩** 시대가 열렸다.
4. 하지만 실무에서는 여전히 품질, 일관성, 검증, 재현성이 문제였고,
5. 그래서 이제는 **하네스 엔지니어링**, 즉 AI가 안정적으로 잘 일할 수 있는 작업 환경 전체를 설계하는 일이 중요해졌다는 것이다.

쉽게 말하면,
**“AI에게 뭘 말할까?”에서 시작해서, 이제는 “AI가 어떤 규칙과 절차 안에서 일하게 만들까?”가 더 중요해졌다**는 얘기다.

## 흐름 요약: AI 엔지니어링의 4단계 진화

### 1. 프롬프트 엔지니어링 시대

초기 생성형 AI 활용은 거의 전부 프롬프트에 집중됐다.
이 시기 핵심 질문은 이런 것이었다.
- 어떻게 질문해야 더 정확한 답이 나오나?
- 어떤 문장을 붙이면 코드를 더 잘 짜주나?
- 역할을 부여하면 결과가 더 좋아지나?

영상에서 언급하는 대표 기법은 다음과 같다.

#### 역할 부여 (Role Prompting)
- “너는 시니어 개발자야”
- “너는 친절한 튜터야”
- “너는 QA 엔지니어야”

이런 식으로 모델의 관점을 지정하면 결과물의 톤과 방향이 달라진다.
이건 지금도 여전히 유효하지만, 단독으로는 한계가 있다.

#### Chain of Thought (CoT)
- 답만 내지 말고 중간 추론 과정을 따라가게 만드는 방식
- 복잡한 문제에서 정확도를 올리는 데 도움을 줌

#### ReAct
- 생각(Reasoning)과 행동(Action)을 번갈아 수행하는 구조
- 도구 사용형 에이전트의 원형 같은 개념
- 오늘날 코딩 에이전트, 툴 호출형 AI의 기초 흐름으로 이어짐

영상은 이 시기를 “AI를 잘 말로 다루는 기술”의 시대라고 본다.
하지만 곧 한계가 드러난다.

### 2. 프롬프트 엔지니어링의 한계

프롬프트를 아무리 잘 써도 다음 문제가 남는다.
- 모델이 프로젝트 전체 문맥을 충분히 모름
- 긴 작업에서는 맥락이 쉽게 손실됨
- 매번 설명을 다시 해야 함
- 같은 요청을 해도 결과 편차가 큼
- 일회성 성공은 가능해도 재현성이 낮음

즉, 문제는 단지 “명령을 잘 쓰는가”가 아니라,
**AI가 어떤 정보 위에서 판단하는가**로 이동한다.

---

### 3. 컨텍스트 엔지니어링 시대

이 단계에서는 프롬프트보다 더 넓은 범위가 중요해진다.
영상은 컨텍스트 엔지니어링을, 모델이 작업할 때 참고하는 **전체 입력 환경을 설계하는 일**로 본다.

주요 구성 요소 예시는 다음과 같다.
- 시스템 프롬프트
- 프로젝트 규칙 파일
- RAG
- MCP 같은 외부 도구 연결
- 필요한 문서와 코드 컨텍스트 주입

핵심은,
AI가 똑똑하지 않아서 못하는 경우보다 **필요한 정보를 못 받아서 못하는 경우**가 훨씬 많다는 점이다.

그래서 좋은 성능은 좋은 문장 하나보다,
- 어떤 파일을 읽히는지
- 어떤 규칙을 우선 적용하는지
- 어떤 외부 지식을 붙이는지
- 어떤 도구를 호출할 수 있는지
에 더 크게 좌우된다.

영상은 프롬프트 엔지니어링과 컨텍스트 엔지니어링을 이렇게 대비한다.

- 프롬프트 엔지니어링: “어떻게 말할까?”
- 컨텍스트 엔지니어링: “무엇을 알고 일하게 할까?”

이건 아주 중요한 전환이다.
왜냐하면 실무 작업은 단발성 질문이 아니라,
문서, 코드, 규칙, 이력, 외부 데이터가 얽힌 **상황 기반 작업**이기 때문이다.

---

### 4. 바이브 코딩의 등장

영상은 다음 단계로 바이브 코딩을 짚는다.

여기서 말하는 바이브 코딩은,
세부 구조를 깊게 설계하지 않아도 AI와 대화하며 빠르게 앱, 페이지, 프로토타입을 만들어내는 흐름이다.

예시 서비스로는 이런 계열이 언급된다.
- Lovable
- Replit
- v0
- Bolt

바이브 코딩의 장점은 엄청 직관적이다.
- 진입장벽이 낮다
- 빠른 결과가 나온다
- 비개발자도 뭔가를 만들 수 있다
- 초기 MVP나 아이디어 검증에 유리하다

하지만 영상은 여기서도 한계를 짚는다.
- 결과물이 빠르지만 구조가 약할 수 있음
- 규모가 커질수록 유지보수가 어려워짐
- 품질 기준이 흔들리기 쉬움
- 실서비스 수준의 안정성, 검증, 일관성이 부족할 수 있음

즉,
**“빨리 만들기”와 “제대로 굴러가게 만들기”는 다른 문제**라는 것이다.

---

### 5. 하네스 엔지니어링의 등장

여기서 영상의 본론이 나온다.

하네스 엔지니어링은 단순한 프롬프트 기법이 아니다.
AI가 반복적으로, 안정적으로, 일관되게 잘 작동하도록 만드는 **작업 시스템 설계**에 가깝다.

핵심 질문은 이렇게 바뀐다.
- AI에게 무슨 말을 할까? → 프롬프트 엔지니어링
- AI에게 뭘 보여줄까? → 컨텍스트 엔지니어링
- AI가 어떤 구조 안에서 일하게 할까? → 하네스 엔지니어링

즉 하네스는 AI를 위한 작업장이다.
좋은 하네스는 다음을 포함한다.
- 역할 분리
- 규칙 파일
- 스킬 패키지
- 훅과 강제 장치
- 검증 루프
- 반복 가능한 워크플로우

영상은 개발자의 역할이 이제 **코드를 직접 많이 치는 사람**에서,
**AI가 잘 일하는 환경을 구축하는 사람**으로 이동하고 있다고 말한다.

이 포인트가 굉장히 중요하다.
앞으로 생산성 차이는 단순히 모델 구독 여부가 아니라,
**누가 더 좋은 하네스를 갖고 있느냐**에서 벌어질 가능성이 크기 때문이다.

---

## 하네스 엔지니어링의 핵심 구성요소

영상에서 설명하는 실전 구성요소는 4가지다.

### 1. 커맨드 (프롬프트 템플릿)

반복 작업을 일관되게 수행하도록 하는 명령 템플릿이다.
예를 들면,
- 버그 수정용 템플릿
- 코드 리뷰용 템플릿
- 문서 생성용 템플릿
- PR 요약용 템플릿

중요한 건 매번 처음부터 설명하지 않는 것이다.
좋은 커맨드는 업무 패턴을 재사용 가능한 형태로 만든다.

### 2. 룰 (규칙 파일)

AI가 항상 따라야 하는 작업 원칙이다.
예를 들면,
- 어떤 파일을 먼저 읽어야 하는지
- 어떤 스타일을 따라야 하는지
- 어떤 작업은 반드시 사용자 확인을 거쳐야 하는지
- 어떤 테스트를 통과해야 완료로 볼지

이 룰이 있으면 결과물의 편차가 줄어든다.
사람 팀에서 팀 컨벤션이 중요한 것과 같은 이유다.

### 3. 스킬

단순 템플릿보다 더 고도화된 작업 묶음이다.
특정 목적을 위해 필요한
- 지침
- 파일 구조
- 스크립트
- 반복 절차
를 하나의 패키지처럼 재사용하는 개념으로 볼 수 있다.

이건 하네스를 점점 시스템화하는 단계다.
한 번 잘 만든 작업 흐름을 여러 프로젝트에 재사용할 수 있게 된다.

### 4. 훅 (강제 보장 장치)

훅은 특히 중요하다.
AI가 까먹거나 임의로 넘기지 못하게, 특정 조건에서 **반드시 실행되게 만드는 장치**다.

예시로 생각해볼 수 있는 건 이런 것들이다.
- 파괴적 명령 전 확인 강제
- 특정 파일 미확인 시 작업 차단
- 테스트 실패 시 종료
- 커밋 전에 체크리스트 실행
- 문서 누락 시 경고

영상은 이 훅을 통해 하네스가 단순한 도움말이 아니라,
**실제로 행동을 강제하는 구조**가 될 수 있다고 보여준다.

## 실제 워크플로우 관점에서의 의미

영상은 하네스가 개념 놀음이 아니라 실전 생산성 도구라는 점을 강조한다.

좋은 워크플로우는 보통 이런 흐름을 갖는다.

1. 작업 유형을 식별한다.
2. 해당 작업에 맞는 커맨드/룰/스킬을 불러온다.
3. AI가 필요한 파일과 컨텍스트를 읽는다.
4. 지정된 형식으로 작업한다.
5. 훅이 빠뜨린 절차를 강제한다.
6. 검증과 리뷰를 거친다.
7. 결과를 문서화하거나 커밋한다.

이 구조의 장점은 명확하다.
- 결과 품질이 덜 흔들린다
- 새 프로젝트에도 복붙이 가능하다
- 협업이 쉬워진다
- 사람이 매번 감독하지 않아도 된다
- AI 활용이 일회성 장난감이 아니라 운영 시스템이 된다

## 개발자 역할의 변화

영상에서 가장 인상적인 주장 중 하나는 이것이다.

**개발자의 역할이 코드 작성 자체에서, 환경 구축과 시스템 설계 쪽으로 이동하고 있다.**

이걸 너무 극단적으로 받아들이면 안 되지만,
방향 자체는 꽤 설득력 있다.

왜냐하면 앞으로는
- 직접 코드 한 줄 한 줄 쓰는 시간보다
- 어떤 구조로 AI를 일시킬지 정의하는 시간이 늘어나고,
- 이 설계 품질이 전체 생산성을 크게 좌우할 가능성이 높기 때문이다.

즉,
좋은 개발자는 이제 단순 구현자가 아니라,
**작업 흐름, 제약, 컨텍스트, 검증 체계를 설계하는 오케스트레이터**에 가까워진다.

---

## 이 영상이 말하는 실전 교훈

### 1. 프롬프트만 잘 써서는 부족하다
실무는 문장 싸움이 아니다.
정보, 구조, 절차, 검증이 다 필요하다.

### 2. 컨텍스트를 잘 먹이는 것이 성능의 큰 부분이다
필요한 문서, 규칙, 코드, 외부 도구를 연결하지 않으면 AI는 계속 헛다리 짚는다.

### 3. 빠른 생성보다 재현 가능한 시스템이 더 중요하다
한 번 잘 되는 것보다, 계속 비슷한 품질로 잘 되게 만드는 것이 진짜 생산성이다.

### 4. 훅과 검증 장치가 있어야 사고가 줄어든다
좋은 AI 활용은 속도만이 아니라, 실패 반경을 줄이는 설계까지 포함한다.

### 5. 결국 경쟁력은 하네스에서 나온다
같은 모델을 써도 누군가는 훨씬 더 안정적으로, 더 빠르게, 더 좋은 결과를 낸다.
그 차이를 만드는 게 하네스다.

## Leo 관점에서 바로 연결되는 포인트

이 영상은 Leo한테 꽤 잘 맞는다.
이유는 Leo가 단일 프로젝트 하나만 보는 사람이 아니라,
여러 프로젝트를 동시에 굴리면서 자동화 수익 구조를 만들려는 타입이기 때문이다.

이런 상황에서는 단순히 “AI가 코드를 좀 써준다” 수준으로는 한계가 빨리 온다.
필요한 건 프로젝트마다 반복 가능한 작업 체계를 만드는 것이다.

예를 들면 바로 이런 식으로 연결된다.
- 프로젝트별 `AGENTS.md`, `SOUL.md`, `USER.md` 같은 운영 규칙
- 역할 분리된 서브에이전트 구조
- 유튜브 요약처럼 반복되는 작업의 템플릿화
- 문서 작성, 구현, 검증, 커밋 흐름의 고정
- 큰 수정 전에 확인받는 안전장치

즉, Leo가 여러 프로젝트를 병렬로 굴릴수록,
프롬프트 스킬보다 **운영 시스템 스킬**이 훨씬 더 큰 레버리지가 된다.

## 추천 액션 아이템

이 영상을 보고 바로 적용할 수 있는 건 다음 정도다.

### 1. 반복 작업을 템플릿으로 고정하기
예:
- 유튜브 노트 생성
- 버그 수정 보고서
- 기능 구현 요청서
- PR 리뷰 요약

### 2. 프로젝트별 규칙 파일 강화하기
예:
- 코딩 컨벤션
- 작업 순서
- 테스트 기준
- 문서화 규칙
- 위험 작업 승인 기준

### 3. 역할 분리하기
예:
- 설계 담당 AI
- 구현 담당 AI
- 리뷰 담당 AI

## 결론

이 영상은 단순히 새로운 유행어 하나를 소개하는 영상이 아니다.
AI 활용이 왜 점점 더 시스템 설계의 문제로 바뀌고 있는지, 그 흐름을 꽤 깔끔하게 설명한다.

핵심 결론은 이렇다.

**앞으로 중요한 건 좋은 모델을 쓰는 것만이 아니라, 그 모델이 잘 일할 수밖에 없도록 만드는 하네스를 갖추는 것이다.**

그리고 그 하네스는 결국,
- 프롬프트
- 컨텍스트
- 규칙
- 도구
- 검증
- 자동화된 절차
를 묶어내는 운영 시스템이다.

한마디로 정리하면,
**AI 시대의 개발 실력은 점점 “잘 코딩하는 능력”만이 아니라 “AI가 잘 일하게 만드는 능력”으로 확장되고 있다.**
