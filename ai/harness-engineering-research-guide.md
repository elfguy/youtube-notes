# 🧭 하네스 엔지니어링 리서치 가이드

> **작성일:** 2026-04-09
> **주제:** Harness Engineering
> **목적:** 하네스 엔지니어링을 자세히 설명하는 사이트, 아티클, 논문, 프리프린트, 서베이 자료를 정리하고, 실무적으로 어떤 순서로 읽으면 좋은지 안내한다.

---

## 한 줄 결론

**하네스 엔지니어링은 아직 완전히 정착된 학문 용어라기보다, 빠르게 부상 중인 실무 개념이자 초기 연구 주제다.**

그래서 지금은 전통적인 교과서나 정식 학회 논문보다,
- 실무 아티클
- 프리프린트
- arXiv 논문
- GitHub 서베이 저장소
- 지식맵/레퍼런스 허브
쪽에 더 좋은 자료가 모여 있다.

즉, “아직 자료가 없나?”가 아니라,
**“분야가 막 이름 붙여지고 체계화되는 중이라 자료의 중심이 논문보다 실무/프리프린트 쪽에 있다”**고 보는 게 맞다.

---

## 하네스 엔지니어링이란 무엇인가

하네스 엔지니어링은 단순히 프롬프트를 잘 쓰는 기술이 아니다.
오히려 **모델이 실제로 안정적이고 반복 가능하게 일하도록 만드는 외부 시스템 전체를 설계하는 일**에 가깝다.

쉽게 말하면 이런 질문들에 답하는 분야다.
- 어떤 규칙 파일을 항상 읽게 할 것인가?
- 어떤 도구를 쓸 수 있게 열어둘 것인가?
- 어떤 정보를 컨텍스트에 넣고 무엇을 뺄 것인가?
- 실패하면 어떻게 복구하게 할 것인가?
- 테스트, 검증, 리뷰는 어디서 어떻게 걸 것인가?
- 여러 에이전트가 있다면 역할을 어떻게 나눌 것인가?

즉,
**모델 자체보다 “모델이 일하는 작업장”을 설계하는 일**이라고 보면 된다.

이 관점에서 많이 쓰이는 요약은 다음과 같다.

> **Agent = Model + Harness**

여기서 harness는 모델을 둘러싼 모든 운영 구조다.
예를 들면,
- 시스템 프롬프트
- 규칙 파일 (`AGENTS.md`, `CLAUDE.md` 등)
- 툴 호출 구조
- 컨텍스트 관리
- 상태 저장
- 재시도 및 복구 로직
- 훅과 가드레일
- 검증 루프
- 평가 인터페이스

결국 하네스 엔지니어링은,
**AI를 “똑똑하게 만드는 기술”이라기보다 “실수하기 어렵게 만드는 환경 설계 기술”**에 가깝다.

---

## 왜 지금 중요해졌는가

생성형 AI 초반에는 프롬프트 엔지니어링이 중심이었다.
하지만 실무에서는 금방 한계가 드러났다.

### 프롬프트만으로는 해결 안 되는 문제
- 긴 작업에서 맥락이 흐려짐
- 프로젝트 고유 규칙을 자주 놓침
- 반복 작업 품질이 흔들림
- 검증 없는 생성 결과는 위험함
- 한 번 잘 된 결과를 다시 재현하기 어려움

그래서 흐름이 이렇게 이동했다.

1. **프롬프트 엔지니어링**
   - 어떻게 말할까?
2. **컨텍스트 엔지니어링**
   - 무엇을 알고 일하게 할까?
3. **하네스 엔지니어링**
   - 어떤 규칙과 절차 안에서 일하게 만들까?

이제 중요한 건 단순히 좋은 답변 한 번이 아니라,
**계속 비슷한 품질로, 통제 가능하게, 실패해도 복구 가능하게 일하는 시스템**이다.

그래서 하네스 엔지니어링은 코딩 에이전트, 브라우저 에이전트, 워크플로우 자동화, 멀티 에이전트 시스템 전반에서 핵심 개념으로 떠오르고 있다.

## 1. Martin Fowler 아티클

### Harness engineering for coding agent users
- 링크: <https://martinfowler.com/articles/harness-engineering.html>
- 성격: 실무 아티클
- 추천도: 매우 높음

이 글은 하네스 엔지니어링을 처음 이해할 때 가장 좋은 입문 자료 중 하나다.

핵심 아이디어는 다음과 같다.

### 1) 하네스는 모델 바깥의 시스템이다
글은 하네스를 AI 에이전트에서 모델을 제외한 나머지 시스템으로 설명한다.
다만 범위를 너무 넓게 잡지 않고, **코딩 에이전트 사용자 입장**에서 유용하게 좁혀서 설명한다.

즉,
- 시스템 프롬프트
- 코드 검색 방식
- 규칙 파일
- 테스트
- 린터
- 코드 리뷰 에이전트
- 구조 검사
같은 것들이 하네스에 포함된다.

### 2) 하네스는 두 가지를 해야 한다
좋은 하네스는 다음 두 가지를 동시에 달성해야 한다고 본다.

- **처음부터 맞게 하도록 유도하기**
- **틀렸을 때 스스로 수정하게 만들기**

이를 글에서는 다음 두 축으로 설명한다.

#### Feedforward controls
사전에 잘못된 행동 가능성을 줄이는 가이드
- 규칙 파일
- 코딩 컨벤션
- 부트스트랩 지침
- 구조적 제약

#### Feedback controls
이미 행동한 뒤 문제를 발견하고 스스로 수정하게 만드는 센서
- 테스트
- 린터
- 타입 체커
- 구조 분석
- AI 코드 리뷰

이 구분이 굉장히 유용하다.
실무에서 “이 문제는 사전 가이드로 막을 건가, 사후 검출로 잡을 건가?”를 생각하게 해주기 때문이다.

### 3) 계산적 제어와 추론적 제어를 나눈다
글은 가이드와 센서를 다시 두 종류로 나눈다.

#### Computational
- deterministic
- 빠름
- CPU 기반
- 테스트, 린터, 타입 체크, 구조 검사

#### Inferential
- semantic
- 느리고 비쌈
- AI 판단 기반
- AI 리뷰, 의미 중복 코드 감지, 판단형 평가

즉,
- 계산적 제어는 자주 돌릴 수 있고 믿을 수 있음
- 추론적 제어는 더 풍부하지만 비싸고 비결정적임

이건 하네스 설계에서 매우 현실적인 프레임이다.
“무엇을 매 커밋마다 돌릴지”, “무엇을 리뷰 단계에서만 돌릴지”를 구분하는 기준이 된다.

### 4) 인간의 역할은 하네스를 개선하는 쪽으로 이동한다
글에서 특히 중요한 문장은,
문제가 반복될 때마다 사람은 단순히 그 작업만 고치는 게 아니라 **하네스를 개선해 같은 문제가 재발하지 않게 해야 한다**는 점이다.

이건 거의 하네스 엔지니어링의 핵심 철학이다.

즉,
- 한 번 생긴 실수는 그냥 넘기지 말고
- 룰, 훅, 테스트, 리뷰 루프 중 어디를 강화할지 보고
- 시스템 차원에서 재발 방지를 설계해야 한다

### 이 글이 좋은 이유
- 실무 언어로 설명함
- 과장 없이 현실적인 구조 제시
- 바로 팀 워크플로우에 적용 가능
- 하네스를 “철학”이 아니라 “운영 시스템”으로 보게 해줌

### 추천 대상
- 코딩 에이전트 실제 사용자
- 팀 단위 AI 개발 프로세스 설계자
- AI 활용 품질 문제로 고생해본 개발자

## 2. OpenAI 아티클

### Harness engineering: leveraging Codex in an agent-first world
- 링크: <https://openai.com/index/harness-engineering/>
- 성격: 실무/엔지니어링 아티클
- 추천도: 높음

이번 리서치에서는 본문 fetch가 403으로 막혔지만, 검색 스니펫만으로도 방향성은 꽤 분명했다.

핵심 메시지는 대략 이 흐름이다.
- 소프트웨어 엔지니어 팀의 주요 일이 더 이상 직접 코드를 쓰는 것만이 아니다.
- 이제는 **환경을 설계하고**, **의도를 명세하고**, **피드백 루프를 만들고**, **에이전트가 신뢰 가능하게 일하도록 시스템을 짜는 것**이 중요해진다.

이 아티클은 특히 Codex 같은 코딩 에이전트를 실제 팀 환경에서 쓸 때,
왜 단순 프롬프트 노하우보다 **repo map, 규칙 파일, 아키텍처 룰, cleanup loop, runtime control** 같은 것들이 중요해지는지를 다룰 가능성이 높다.

검색 결과 기준으로 보면,
이 자료는 “하네스 엔지니어링”이라는 단어를 OpenAI 맥락에서 실무적으로 밀고 있는 대표 자료 중 하나로 보인다.

### 추천 이유
- 코딩 에이전트 실전 맥락과 연결됨
- “직접 코딩”에서 “환경 설계”로의 역할 이동을 설명함
- 대형 리포지토리, 장기 작업, 다단계 루프에 유용한 관점을 줄 가능성이 큼

### 한계
- 본문 직접 수집이 안 되어 이번 문서에서는 2차 정보 수준으로만 정리함
- 나중에 직접 읽고 별도 업데이트하는 게 좋음

## 3. arXiv 논문

### Meta-Harness: End-to-End Optimization of Model Harnesses
- 링크: <https://arxiv.org/abs/2603.28052>
- 성격: 연구 논문 (arXiv)
- 추천도: 매우 높음

이 논문은 현재 확인한 자료 중 **가장 직접적으로 “하네스”를 연구 대상으로 삼는 논문** 중 하나다.

초록 기준 핵심은 다음과 같다.

### 논문이 던지는 문제의식
LLM 시스템의 성능은 모델 가중치만으로 결정되지 않는다.
무엇을 저장하고,
무엇을 검색하고,
무엇을 모델에게 보여주는지를 결정하는 **하네스 코드**가 큰 영향을 준다.

그런데 지금까지 이 하네스는 대부분 사람 손으로 설계되어 왔다.
즉,
모델은 열심히 최적화하면서도,
정작 하네스는 감과 경험에 많이 의존해 만든 셈이다.

### 논문이 제안하는 것
이 논문은 **Meta-Harness**라는 outer-loop 시스템을 제안한다.
핵심은 하네스 코드를 자동으로 탐색하면서 더 나은 구성을 찾는 것이다.

설명상 특징은 이렇다.
- agentic proposer 사용
- 이전 후보들의 소스 코드, 점수, 실행 추적을 파일시스템으로 접근
- 그 경험을 바탕으로 다음 하네스를 제안

즉,
AI를 잘 쓰기 위한 하네스를 사람만 설계하는 게 아니라,
**하네스 설계 자체도 에이전트적으로 최적화하자**는 방향이다.

### 보고된 성과
초록 기준으로는 다음이 언급된다.
- 온라인 텍스트 분류에서 기존 context management 시스템 대비 **7.7점 개선**
- **컨텍스트 토큰 4배 절감**
- retrieval-augmented math reasoning에서 성능 향상
- agentic coding에서 hand-engineered baseline보다 우수

### 왜 중요한가
이 논문이 중요한 이유는,
단순 선언이 아니라 **“하네스가 실제로 성능을 좌우한다”**는 점을 실험적으로 밀어주기 때문이다.

그리고 한 단계 더 나아가,
하네스 엔지니어링을 앞으로는 수작업 기술이 아니라 **최적화 대상**으로 볼 수 있음을 시사한다.

### 추천 대상
- 하네스 엔지니어링을 개념이 아니라 연구 주제로 보고 싶은 사람
- 컨텍스트 관리, 메모리, retrieval, agent loop 최적화에 관심 있는 사람
- “모델보다 시스템 설계가 중요할 수 있다”는 주장을 근거 있게 보고 싶은 사람

## 4. Preprints 포지션 페이퍼

### Harness Engineering for Language Agents: The Harness Layer as Control, Agency, and Runtime
- 링크: <https://www.preprints.org/manuscript/202603.1756/v1>
- 성격: 포지션 페이퍼 / 프리프린트
- 추천도: 높음
- 주의: **peer review 전 자료**

이 글은 하네스 엔지니어링을 학술적으로 명명하고 구조화하려는 시도가 강하다.

초록 기준 핵심은 다음과 같다.

### 1) 에이전트 신뢰성은 모델만으로 설명되지 않는다
언어 에이전트가 도구, 파일, 브라우저, API, 세션을 다루기 시작하면,
신뢰성은 더 이상 모델이나 단일 프롬프트만으로 설명할 수 없다.

중요한 건 그 바깥 레이어다.
즉,
- 어떤 지시가 계속 권위를 갖는지
- 어떤 액션이 허용되는지
- 상태를 어떻게 외부화하고 이어가는지
- 실패를 어떻게 다루는지

이걸 통틀어 harness layer라고 본다.

### 2) CAR 프레임 제안
이 프리프린트는 harness layer를 다음 3개 축으로 나눈다.

#### Control
무엇이 기준이 되고 무엇이 구속력을 가지는가
- 시스템 규칙
- 정책
- 우선순위
- 권한 구조

#### Agency
에이전트가 실제로 어떤 행동을 할 수 있는가
- 툴 사용
- 액션 인터페이스
- 브라우저, 파일, API 접근
- 멀티스텝 작업 범위

#### Runtime
시간에 걸쳐 어떻게 실행되고 관리되는가
- 상태 유지
- 실패 복구
- 로그
- 검사 가능성
- bounded execution

이 CAR 프레임은 아직 표준은 아니지만,
하네스를 정리하는 언어로는 꽤 쓸만하다.

### 3) 연구 보고에도 하네스를 같이 써야 한다는 주장
이 논문의 중요한 주장 중 하나는,
앞으로 언어 에이전트 성능을 말할 때 **모델만 보고하지 말고 하네스도 같이 보고해야 한다**는 것이다.

이 말은 굉장히 설득력 있다.
실제로 같은 모델이어도
- 어떤 도구를 열어뒀는지
- 어떤 retrieval을 붙였는지
- 어떤 상태 저장 방식을 썼는지
- 어떤 검증 루프가 있었는지
에 따라 결과가 크게 달라질 수 있기 때문이다.

### 이 자료의 가치
- 개념을 학술적 언어로 정리함

### 제안하는 6개 구성요소
리포는 harness를 다음 튜플로 정리한다.

#### 1. Execution Loop (E)
- observe-think-act 루프
- 종료 조건
- 에러 복구

#### 2. Tool Registry (T)
- 툴 카탈로그
- 라우팅
- 스키마 검증
- 모니터링

#### 3. Context Manager (C)
- 어떤 정보가 context window에 들어가는지
- 압축/정리
- retrieval

#### 4. State Store (S)
- 턴/세션 간 지속성
- 충돌 복구
- 상태 저장

#### 5. Lifecycle Hooks (L)
- 인증
- 로깅
- 정책 강제
- 계측

#### 6. Evaluation Interface (V)
- 행동 궤적
- 중간 상태
- 성공 신호

이 프레임은 실무적으로도 매우 유용하다.
왜냐하면 팀이나 프로젝트의 에이전트 구조를 볼 때,
“우리는 이 여섯 개 중 어디가 강하고 어디가 비어 있지?”를 체크하는 데 바로 쓸 수 있기 때문이다.

### 리포가 주장하는 중요한 포인트
리포 설명에는 이런 강한 메시지가 있다.
- agent reliability의 핵심은 모델보다 execution harness에 있다
- 벤치마크 성능과 실제 운영 신뢰성 사이에는 큰 차이가 있다
- 하네스 설계는 배포 환경에서 병목이자 핵심 제약이다

## Leo 관점에서 실무적으로 중요한 포인트

Leo처럼 여러 프로젝트를 동시에 굴리고,
AI를 단순 보조가 아니라 **생산 시스템의 일부**로 쓰려는 경우,
하네스 엔지니어링은 거의 필수 개념에 가깝다.

왜냐하면 프로젝트가 늘어날수록 문제는 모델 성능 자체보다 이런 데서 터지기 때문이다.
- 매번 설명 반복
- 컨텍스트 누락
- 프로젝트별 규칙 미준수
- 검증 누락
- 서브에이전트 간 책임 모호
- 한 번 성공한 흐름이 재현되지 않음

즉,
프롬프트보다 더 중요한 건 다음이다.
- 프로젝트 규칙 파일
- 역할 분리
- 자동 검증 루프
- 승인 기준
- 상태 저장
- 반복 가능한 문서화

이건 곧 다음 질문으로 이어진다.
- 어떤 파일을 항상 읽게 할 것인가?
- 어떤 작업은 승인 필수로 막을 것인가?
- 어떤 테스트는 자동으로 돌릴 것인가?
- 어떤 역할은 분리할 것인가?
- 어떤 실패는 자동 복구하게 할 것인가?

이게 바로 하네스 설계다.

## 실전 적용 체크리스트

하네스 엔지니어링을 실제 프로젝트에 적용할 때는 아래 항목을 보면 된다.

### Control
- 시스템 규칙이 명확한가?
- 우선순위가 충돌하지 않는가?
- 작업 금지선이 정의돼 있는가?

### Agency
- 툴 권한이 적절한가?
- 할 수 있는 액션 범위가 과하지 않은가?
- 역할 분리가 되어 있는가?

### Runtime
- 상태를 어디에 저장하는가?
- 실패했을 때 복구 루프가 있는가?
- 장기 작업 로그와 진행상태가 남는가?

### Verification
- 테스트, 린터, 리뷰 훅이 있는가?
- AI가 자기 수정 가능한 피드백 루프가 있는가?
- 사람이 최종 확인해야 할 지점이 분명한가?

### Reuse
- 반복 작업이 템플릿화되어 있는가?
- 스킬/커맨드/워크플로우가 재사용 가능한가?
- 프로젝트가 바뀌어도 복제 가능한 구조인가?

## 최종 정리

하네스 엔지니어링은 아직 완전히 제도화된 학문이라기보다는,
**실무 현장에서 먼저 필요성이 폭발했고, 이제 그걸 연구와 이론이 따라가는 단계**에 있다.

그래서 현재 가장 좋은 접근은 다음과 같다.

1. **실무 아티클로 감을 잡고**
2. **논문/프리프린트로 구조를 이해하고**
3. **서베이/지식맵으로 확장하는 것**

핵심 메시지를 한 문장으로 정리하면 이렇다.

**AI 에이전트의 성능은 모델만으로 결정되지 않는다. 실제 성능과 신뢰성은 모델을 둘러싼 하네스, 즉 규칙, 도구, 상태, 검증, 실행 구조에 의해 크게 좌우된다.**

그래서 앞으로 경쟁력은 단순히 좋은 모델을 쓰는 데서 끝나지 않고,
**그 모델이 잘 일할 수밖에 없도록 만드는 운영 시스템을 얼마나 잘 설계하느냐**에서 크게 갈릴 가능성이 높다.
