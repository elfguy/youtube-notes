## 핵심 메시지

**"죽은 건 RAG가 아닙니다. RAG를 단순하게 쓰던 시절이 끝난 겁니다."**

코딩 에이전트에서는 RAG 없이도 가능하지만, 대부분의 AI 시스템(기업용, 문서검색, 챗봇 등)에서는 **RAG 없이 프로덕션을 돌릴 수 없다**.

---

## 📋 영상 구조 (타임스탬프)

### 1. RAG는 정말 죽었을까? (0:00)
- "RAG is dead" 논쟁 소개
- 결론: 죽지 않았음, 진화 중

### 2. RAG란 무엇인가, 왜 필요한가 (2:02)
- **RAG = Retrieval-Augmented Generation** (검색 증강 생성)
- LLM은 학습 데이터 이후 정보를 모름 → 외부 데이터를 검색해서 LLM에 전달
- 기업 내부 문서, 최신 정보, 도메인 특화 지식에 필수

### 3. Embedding과 벡터: 텍스트를 숫자로 바꾸는 원리 (4:24)
- **Embedding** = 텍스트를 숫자 벡터로 변환
- 의미가 비슷한 텍스트 → 벡터 공간에서 가까이 위치
- 이 벡터로 "의미 기반 검색" 가능

### 4. 청킹과 Parser: 문서를 어떻게 잘라야 검색이 되는가 (6:30)
- **청킹(Chunking)** = 긴 문서를 적절한 단위로 자르는 것
- 너무 크면 노이즈 ↑, 너무 작으면 맥락 ↓
- **Parser** = PDF, 표, 이미지 등 다양한 문서 형식을 텍스트로 변환
- **"청크가 정크(junk)가 되면 RAG도 정크"** — 파싱/청킹 품질이 핵심

### 5. 벡터 데이터베이스: 숫자를 저장하고 검색하는 방법 (8:19)
- **벡터 DB** = 임베딩 벡터를 저장하고 유사도 검색하는 전용 DB
- 주요 서비스: Pinecone, Weaviate, Qdrant, Chroma, pgvector(Postgres)

### 6. RAG 파이프라인 전체 흐름 정리 (9:23)
```
문서 → Parser → 청킹 → Embedding → 벡터 DB에 저장
                                          ↓
사용자 질문 → Embedding → 벡터 DB 검색 → 관련 청크 추출 → LLM에 전달 → 답변
```

### 7. 실무에서 RAG가 실패하는 진짜 원인 (9:57)
- ❌ 모델이 나빠서가 아님
- ✅ **검색(Retrieval) 단계에서 관련 문서를 못 찾아서**
- 파싱 품질, 청킹 전략, 임베딩 모델 선택이 실패 원인의 대부분

### 8. Dense, Sparse Embedding과 하이브리드 서치 (10:49)
- **Dense Embedding** = 의미 기반 검색 (BERT, OpenAI 등)
  - 장점: 의미적 유사성 포착
  - 단점: 전문용어/고유명사에 약함
- **Sparse Embedding** = 키워드 기반 검색 (BM25, TF-IDF)
  - 장점: 정확한 키워드 매칭
  - 단점: 의미적 유사성 못 잡음
- **하이브리드 서치** = Dense + Sparse 결합
  - 실무에서 가장 효과적인 방법
  - 두 방식의 장점을 동시에 활용

### 9. Reranker: 검색 결과를 정밀하게 재평가하기 (13:06)
- 1차 검색으로 후보를 넓게 가져온 뒤
- **Reranker**가 질문과 문서의 관련성을 정밀 평가 → 순위 재정렬
- Cross-encoder 기반으로 더 정확한 관련성 판단

### 10. 고급 기법: Contextual Retrieval (15:09)
- Anthropic이 제안한 방법
- 각 청크에 "이 청크가 전체 문서에서 어떤 맥락인지" 설명을 추가
- LLM이 청크를 전처리해서 맥락 정보를 덧붙임
- 검색 정확도 대폭 향상

### 11. 고급 기법: Late Chunking (17:17)
- 기존: 먼저 청킹 → 각 청크를 개별 임베딩
- Late Chunking: 전체 문서를 먼저 임베딩 → 그 다음 청킹
- 문서 전체 맥락을 보존한 상태에서 청크 임베딩 생성
- 청크 간 맥락 손실 방지

### 12. 전체 개념 총정리 (18:41)
```
[인덱싱 단계]
문서 → Parser → 청킹 → (Contextual Retrieval) → Embedding (Dense+Sparse) → 벡터 DB

[검색 단계]
질문 → Embedding → 하이브리드 서치 → Reranker → Top-K 청크 → LLM → 답변
```

### 13. "RAG is dead" 논쟁의 진짜 맥락 (19:37)
- **맥락:** LLM의 컨텍스트 윈도우가 커지면서 (100K+ 토큰) "문서 전체를 넣으면 되지 않나?"
- **반론:**
  - 비용: 매 질문마다 문서 전체를 넣으면 API 비용 폭발
  - 속도: 긴 컨텍스트 = 느린 응답
  - 정확도: 긴 컨텍스트에서 LLM이 "중간을 건너뛰는" 현상 (Lost in the Middle)
  - 스케일: 문서가 수천~수만 개면 컨텍스트에 다 못 넣음
- **결론:** RAG는 죽지 않았고, 오히려 더 정교해지고 있음

### 14. 결론: RAG를 배워야 하는 이유 (23:22)
- AI 시스템의 대부분은 외부 데이터 연동이 필요
- RAG는 프로덕션 AI의 기본 아키텍처
- 제조/금융/법률/의료 등 도메인 특수성이 강한 산업에서 필수

## 🎯 이런 분이 보면 좋음

- RAG 파이프라인 전체 구조가 머릿속에 안 잡히는 분
- AI 챗봇/사내 검색 시스템을 만들어야 하는 분
- Embedding, 벡터 DB, 청킹 같은 용어가 따로따로 떠다니는 분
- RAG를 구축했는데 검색 결과가 엉뚱하게 나오는 분
- AI 에이전트를 만들면서 검색 아키텍처 선택 기준을 갖고 싶은 분

## 💡 핵심 인사이트 (댓글 포함)

1. **파싱/청킹이 RAG 성패의 80%** — 모델 성능보다 데이터 전처리가 중요
2. **하이브리드 서치(Dense+Sparse) + Reranker** 조합이 실무 표준
3. **Contextual Retrieval** — 맥락을 잃지 않는 청킹의 핵심 기법
4. **DAU 10만 이하 서비스에서는 Postgres(pgvector)로 통합**하는 게 실용적 (댓글 토론)
5. **문서 타입별 청킹 정책 문서를 먼저 만들고**, 도메인 전문가 피드백으로 품질 확보 (실무자 댓글)
