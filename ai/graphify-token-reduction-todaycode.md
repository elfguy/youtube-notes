## 한 줄 요약

안드레이 카파시가 제안한 LLM 위키 개념에서 출발한 **Graphify**라는 도구가, 기존 RAG의 벡터 임베딩 방식을 전혀 쓰지 않고 **파이썬 코드로 그래프를 생성한 뒤 결정론적으로 실행한 결과를 LLM에 제공**하는 방식으로, 토큰 비용을 71.5배 절감하면서도 정확도를 높이는 원리를 분석한 영상이다.

---

## 핵심 메시지

1. **Graphify는 벡터 임베딩도, 벡터 DB도 안 쓴다.** 대신 파이썬 코드로 그래프를 생성하고, 그걸 실행한 결과를 LLM에 문맥으로 제공한다.

2. **LLM이 모든 걸 추론하는 게 아니라, 코드가 처리할 수 있는 건 코드에 맡긴다.** 이게 비용 절감과 정확도 향상의 핵심이다.

3. **71.5배 토큰 절감**이 가능한 이유는, 임베딩 모델 호출도 벡터 DB도 필요 없이, 결정론적 코드 실행 결과만 LLM에 전달하기 때문이다.

4. 카파시의 "LLM 위키" 개념 — LLM이 필요한 정보를 위키처럼 탐색해서 가져오는 방식 — 을 실제 도구(Graphify)로 구현한 사례를 분석한다.

## 배경: 안드레이 카파시의 LLM 위키 개념

영상의 출발점은 안드레이 카파시(Andrej Karpathy)가 제안한 개념이다.

### 카파시의 아이디어
LLM에게 모든 문서를 한 번에 넣어주는 게 아니라, LLM이 **위키처럼 필요한 정보를 능동적으로 탐색**해서 가져오게 만들자는 발상이다.

이건 기존 RAG와 비슷해 보이지만, 접근 방식이 다르다.
- 기존 RAG: 임베딩 → 벡터 검색 → 상위 k개 문서 청크를 컨텍스트에 삽입
- 카파시 방식: LLM이 직접 탐색 경로를 결정하고, 필요한 만큼만 정보를 가져옴

### 왜 이게 중요한가
기존 RAG의 문제점:
- 임베딩 품질에 결과가 크게 좌우됨
- 벡터 검색이 의미적으로 항상 정확하지 않음
- 컨텍스트에 불필요한 청크가 많이 들어감 → 토큰 낭비
- 임베딩 모델 + 벡터 DB 운영 비용 발생

카파시 방식은 이 문제를 다른 각도에서 공격한다.

## Graphify란 무엇인가

Graphify는 이 카파시 개념을 실제로 구현한 도구다.

### 핵심 원리
1. **파이썬 코드로 그래프를 먼저 생성**한다
   - 문서/코드/데이터의 구조와 관계를 그래프 형태로 표현
   - 이 그래프는 임베딩이 아니라 코드로 생성됨 (결정론적)

2. **그래프를 결정론적으로 실행**한다
   - LLM이 추론하는 게 아니라, 파이썬 코드가 실행
   - 실행 결과가 문서 형태로 나옴

3. **실행 결과 문서를 LLM과 함께 활용**한다
   - LLM은 이 결과를 읽고 답변을 생성
   - 불필요한 정보가 걸러진 상태라 토큰 사용이 효율적

### 기존 RAG와의 차이

| 항목 | 기존 RAG | Graphify |
|---|---|---|
| 정보 검색 | 벡터 임베딩 + 유사도 검색 | 파이썬 코드로 그래프 생성 후 실행 |
| 임베딩 모델 | 필요함 | 불필요 |
| 벡터 DB | 필요함 | 불필요 |
| 토큰 사용 | 많음 (불필요한 청크 포함) | 적음 (필요한 결과만 전달) |
| 결정성 | 비결정적 (유사도 기반) | 결정론적 (코드 실행 기반) |
| 비용 | 임베딩 + DB + LLM 토큰 | LLM 토큰만 |

## 71.5배 토큰 절감의 원리

영상 제목의 "71.5배" 숫자는 이렇게 나온다.

### 기존 방식의 토큰 소모
- 전체 문서를 컨텍스트에 넣거나
- RAG로 검색한 여러 청크를 컨텍스트에 넣음
- 이 과정에서 불필요한 정보가 많이 포함됨
- 임베딩 모델 호출 비용도 별도

### Graphify 방식의 토큰 소모
- 코드가 실행된 결과만 LLM에 전달
- 필요한 정보만 정확하게 포함
- 임베딩 호출 없음
- 벡터 DB 오버헤드 없음

### 왜 71.5배인가
기존 방식에서 수십만 토큰을 써야 할 작업이,
Graphify로는 수천 토큰 수준으로 줄어든다는 의미다.

이건 단순히 "문서를 요약해서 넣는다"가 아니라,
**코드가 미리 걸러내고 계산한 결과만 LLM에게 준다**는 구조적 차이에서 나온다.

## 검증 실험 내용

영상에서는 두 가지 실제 케이스로 Graphify를 검증한다.

### 실험 1: 대한민국 법령 문서 분석
- **데이터:** legalize-kr 저장소의 법령 문서
- **목적:** 방대한 법령 텍스트에서 특정 질문에 정확하게 답하기
- **결과:** 기존 방식 대비 토큰 절감 + 정확도 향상 확인

이 실험의 의미:
- 법령은 구조화되어 있지만 양이 방대함
- 기존 RAG로는 관련 조문을 정확히 찾기 어려움
- Graphify는 법령의 구조를 그래프로 표현하고, 필요한 조문만 코드로 추출

### 실험 2: Graphify 자체 코드베이스 분석
- **데이터:** Graphify 저장소 전체 코드
- **목적:** Graphify 자체의 코드를 Graphify로 분석 (메타 분석)
- **결과:** 자기 자신의 코드도 잘 분석함 → 도구의 범용성 확인

## 핵심 인사이트

### 1. 임베딩 없이도 정확한 검색이 가능하다
벡터 임베딩이 정보 검색의 유일한 방법이 아니다.
코드로 구조를 표현하고 실행하는 방식이 더 정확할 수 있다.

### 2. 결정론적 실행이 LLM 추론보다 저렴하고 정확하다
LLM이 "이 문서에서 찾아줘"라고 추론하는 것보다,
파이썬 코드가 "이 조건에 맞는 데이터를 추출해"라고 실행하는 게 더 빠르고 정확하다.

### 3. LLM의 역할을 재정의해야 한다
LLM이 모든 걸 다 하는 게 아니라,
- 코드가 할 수 있는 건 코드에 맡기고
- LLM은 결과를 해석하고 답변을 생성하는 데만 쓴다

이게 하네스 엔지니어링의 관점과도 연결된다.
좋은 하네스는 LLM에게 맞는 역할만 주는 구조다.

### 4. 토큰 비용은 구조 설계로 줄일 수 있다
모델을 바꾸거나 파인튜닝하지 않아도,
**어떤 정보를 어떻게 LLM에게 전달하느냐**만 바꿔도 비용이 71.5배 줄어든다.

## Leo 관점에서 중요한 포인트

### OWL / PI 시그널에 적용 가능한 인사이트
Leo가 운영하는 코인 자동매매 시스템에서도,
시장 데이터, 뉴스, 지표 등을 LLM에 전달할 때 같은 원리가 적용될 수 있다.

- **현재:** raw 데이터나 전체 컨텍스트를 LLM에 넣을 가능성
- **개선 방향:** 코드로 먼저 걸러내고 계산한 결과만 LLM에 전달
- **효과:** API 비용 절감 + 응답 정확도 향상 + 속도 개선

### 일반적 적용
- 어떤 문서/코드/데이터를 LLM에 넣을 때
- 전체를 넣지 말고, 코드로 전처리한 결과만 넣기
- 이건 RAG를 쓰든 Graphify를 쓰든 적용되는 원칙

### 하네스 엔지니어링과의 연결
Graphify의 방식은 좋은 하네스의 예시다.
- LLM이 직접 검색하지 않고 코드가 대신 검색 (역할 분리)
- 결정론적 실행으로 비결정적 LLM 추론을 보완 (검증 루프)
- 필요한 정보만 LLM에 전달 (컨텍스트 최적화)
