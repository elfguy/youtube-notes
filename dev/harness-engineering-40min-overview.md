## 핵심 메시지

**"좋은 결과는 더 좋은 프롬프트보다, 더 좋은 하네스에서 나온다."**

이 영상은 Claude Code와 Codex를 함께 쓰는 흐름 속에서 **하네스 엔지니어링**이 왜 중요한지 40분 동안 개념적으로 정리하는 내용이다.

핵심은 단순하지만 꽤 무겁다.
- AI 하나에게 다 시키는 방식은 금방 한계가 온다.
- 같은 모델이라도 어떤 규칙, 어떤 순서, 어떤 검증 루프 안에서 일시키느냐에 따라 결과가 크게 달라진다.
- 그래서 이제 중요한 건 “프롬프트를 얼마나 잘 쓰느냐”보다 “AI가 잘 일할 수밖에 없게 환경을 어떻게 짜느냐”다.

이걸 개발자 관점으로 풀면 이런 느낌이다.
- 좋은 개발자는 함수를 잘 짜는 사람만이 아니라, 시스템을 잘 설계하는 사람이다.
- 좋은 AI 활용자도 프롬프트 장인이 아니라, **작업 구조를 설계하는 사람**으로 바뀌고 있다.

즉 이 영상의 메시지는, AI를 똑똑한 인턴처럼 대하는 수준에서 끝나지 말고, **역할과 절차와 검증이 있는 팀 시스템 안에 넣어야 한다**는 쪽에 가깝다.

## 영상에서 다루는 핵심 주제

### 1. 하네스 엔지니어링이란
- 프롬프트 한 줄 잘 쓰는 문제가 아님
- AI가 일하는 **전체 작업 환경**을 설계하는 것
- 예: 역할 분리, 규칙 파일, 검증 루프, 실행 순서, 안전장치

이 개념을 더 풀어보면, 하네스는 단순한 프롬프트 템플릿이 아니다.
오히려 아래를 모두 포함하는 상위 개념에 가깝다.
- 어떤 파일을 항상 먼저 읽는지
- 어떤 역할이 어떤 책임을 갖는지
- 어떤 순서로 작업을 진행하는지
- 언제 멈추고 사람 확인을 받는지
- 무엇을 테스트하고 어떤 기준으로 통과시키는지

즉, AI의 "두뇌"만 보는 게 아니라 **일하는 작업장 전체**를 설계하는 개념이다.

### 2. Claude Code × Codex 조합
- 한 모델만 고집하기보다 역할별로 더 잘하는 모델을 배치
- 예:
  - **Claude Code**: 큰 그림, 구조화, 긴 컨텍스트, 설계
  - **Codex**: 빠른 수정, 코드 생성, 반복 구현
- 중요한 건 모델 자체보다 **누가 언제 무엇을 맡느냐**

이 부분이 특히 실무적이다.
많은 사람이 "어느 모델이 최고냐"를 묻는데, 실제 생산성은 모델 1등 결정전보다 **책임 분배**에서 나온다.

예를 들면:
- 설계 초안은 Claude가 잡고
- 반복적인 파일 수정은 Codex가 하고
- 마지막 리뷰는 다시 Claude나 다른 관점 에이전트가 보는 식이다.

이때 핵심은 모델 혼용 자체가 아니라, **혼용 기준이 명확해야 한다**는 점이다.
막 섞으면 더 혼란스럽고, 역할이 고정되면 팀처럼 움직인다.

### 3. 계획 → 구현 → 리뷰 분리
- 기획과 코딩과 검증을 한 대화창에서 다 처리하면 품질이 흔들림
- 먼저 계획을 굳히고
- 그다음 구현하고
- 마지막에 별도 관점으로 리뷰해야 함
- 이 구조가 실제 팀 일하는 방식과 닮아 있음

이건 영상의 가장 중요한 포인트 중 하나다.
같은 창에서 "기획도 해줘, 코드도 짜줘, 검토도 해줘"를 한 번에 던지면 AI는 얼핏 그럴듯한 결과를 내지만, 실제로는 각 단계가 섞여서 허술해진다.

왜냐하면:
- 계획 단계는 탐색과 선택이 중요하고
- 구현 단계는 정확성과 일관성이 중요하고
- 리뷰 단계는 의심하고 깨는 시선이 중요하기 때문이다.

한 에이전트가 이 세 가지 모드를 동시에 잘하기는 어렵다.
그래서 이 영상은 결과적으로 **분업 구조가 품질을 만든다**는 말을 하고 있다.

### 4. 재사용 가능한 시스템 만들기
- 매번 새로 설명하지 말고
  - `CLAUDE.md`
  - `SKILL.md`
  - 작업 템플릿
  - 역할별 프롬프트/규칙
  이런 걸 축적해야 함
- 즉, **일회성 대화**보다 **반복 가능한 작업 시스템**이 더 중요

이 부분은 Leo 스타일에 특히 중요하다.
프로젝트가 많아질수록 기억에 의존하면 무조건 무너진다.
그래서 작업 규칙을 파일로 남기고, AI가 매번 그걸 읽고 들어가게 해야 한다.

재사용 가능한 시스템이 생기면 좋은 점:
- 매번 설명 반복 안 해도 됨
- 결과 품질 편차가 줄어듦
- 프로젝트가 늘어나도 관리가 쉬움
- 사람이 바뀌어도 흐름이 유지됨

쉽게 말하면, 일 잘하는 사람의 비결을 머릿속에만 두지 말고 **운영 매뉴얼로 굳히는 것**이다.

### 5. 검증 루프와 안전장치
- 생성만 잘하는 AI는 위험함
- 반드시
  - 리뷰 단계
  - 테스트 단계
  - 범위 제한
  - 실패 시 되돌릴 수 있는 구조
  가 필요함

이 영상이 단순히 "AI를 더 세게 돌리자"는 얘기가 아니라 괜찮은 이유가 여기 있다.
좋은 하네스는 생성 속도를 높이는 동시에 **사고 반경도 줄인다**.

예를 들면:
- 파괴적 수정 전에는 사용자 승인 받기
- 큰 변경은 작은 커밋 단위로 쪼개기
- 테스트 실패 시 자동 중단하기
- QA 역할을 구현 역할과 분리하기

이런 식으로 해야 AI가 생산성을 올리면서도 위험을 통제할 수 있다.

## 조금 더 자세한 해석

### 왜 이 개념이 지금 뜨는가
예전에는 "프롬프트를 잘 쓰면 된다"는 분위기가 강했다.
그런데 실제 현업에서는 금방 부딪힌다.
- 대화가 길어질수록 맥락이 흐려지고
- 같은 요청도 결과 편차가 크고
- 한 번 잘돼도 다음에 다시 재현하기 어렵다.

그래서 이제는 프롬프트보다 **프로세스 재현성**이 더 중요해졌다.
즉, 우연히 잘 되는 한 번보다 **계속 비슷하게 잘 되는 구조**가 더 가치 있어진 거다.

### 하네스의 본질은 ‘역할 전환’이다
AI는 똑같은 모델이어도 역할을 달리 주면 산출물의 결이 바뀐다.
- 제품 기획자처럼 생각하게 할 수도 있고
- 시니어 엔지니어처럼 비판하게 할 수도 있고
- QA처럼 깨는 데 집중하게 할 수도 있다.

결국 하네스는 모델의 지능을 높이는 기술이라기보다, **지능을 올바른 자세로 앉히는 기술**에 가깝다.

### Claude Code와 Codex를 같이 쓰는 이유
이 영상 제목에 굳이 두 도구가 같이 들어간 건 상징적이다.
메시지는 "둘 중 누가 더 좋냐"보다, **둘을 같은 팀 안에서 어떻게 배치하느냐**에 있다.

실무에서는 이런 판단이 중요하다.
- 긴 문맥 이해가 필요한 설계는 누가 맡을지
- 반복 수정, 작은 구현, 대량 수정은 누가 맡을지
- 최종 판단은 누가 할지

이게 정해져 있으면 혼란이 줄고, 안 정해져 있으면 매번 감으로 일하게 된다.
