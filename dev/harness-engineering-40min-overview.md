## 핵심 메시지

**"좋은 결과는 더 좋은 프롬프트보다, 더 좋은 하네스에서 나온다."**

이 영상은 Claude Code와 Codex를 함께 쓰는 흐름 속에서 **하네스 엔지니어링**이 왜 중요한지 40분 동안 개념적으로 정리하는 내용으로 보인다.

핵심은 단순하다.
- AI 하나에게 다 시키는 게 아니라
- 역할을 나누고
- 계획, 구현, 검증을 분리하고
- 반복 가능한 작업 환경을 설계해야
  결과 품질이 올라간다.

---

## 영상에서 다루는 핵심 주제

### 1. 하네스 엔지니어링이란
- 프롬프트 한 줄 잘 쓰는 문제가 아님
- AI가 일하는 **전체 작업 환경**을 설계하는 것
- 예: 역할 분리, 규칙 파일, 검증 루프, 실행 순서, 안전장치

### 2. Claude Code × Codex 조합
- 한 모델만 고집하기보다 역할별로 더 잘하는 모델을 배치
- 예:
  - **Claude Code**: 큰 그림, 구조화, 긴 컨텍스트, 설계
  - **Codex**: 빠른 수정, 코드 생성, 반복 구현
- 중요한 건 모델 자체보다 **누가 언제 무엇을 맡느냐**

### 3. 계획 → 구현 → 리뷰 분리
- 기획과 코딩과 검증을 한 대화창에서 다 처리하면 품질이 흔들림
- 먼저 계획을 굳히고
- 그다음 구현하고
- 마지막에 별도 관점으로 리뷰해야 함
- 이 구조가 실제 팀 일하는 방식과 닮아 있음

### 4. 재사용 가능한 시스템 만들기
- 매번 새로 설명하지 말고
  - `CLAUDE.md`
  - `SKILL.md`
  - 작업 템플릿
  - 역할별 프롬프트/규칙
  이런 걸 축적해야 함
- 즉, **일회성 대화**보다 **반복 가능한 작업 시스템**이 더 중요

### 5. 검증 루프와 안전장치
- 생성만 잘하는 AI는 위험함
- 반드시
  - 리뷰 단계
  - 테스트 단계
  - 범위 제한
  - 실패 시 되돌릴 수 있는 구조
  가 필요함
