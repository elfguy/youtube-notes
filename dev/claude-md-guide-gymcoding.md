  - 자기 검증 규칙을 넣어야 실효성 있음

### 3. 오토 메모리(Auto Memory) 최신 스펙 (4:01)

### 4. ❌ 나쁜 CLAUDE.md 패턴 3가지

**패턴 ① 너무 긴 파일 (5:42)**
- 200줄 초과하면 Claude가 핵심을 놓침
- 긴 파일 = 노이즈 ↑ = 품질 ↓

**패턴 ② 모호한 지시 (6:35)**
- "깔끔하게 코드 작성해줘" → 검증 불가능
- "변수명은 camelCase로" → 검증 가능 ✅

**패턴 ③ 프로젝트 커져도 파일 하나로 버티기 (7:14)**
- 프로젝트가 커지면 모듈별로 분리해야 함
- 하나의 거대한 CLAUDE.md = 관리 불가능

### 5. ✅ 좋은 CLAUDE.md 작성법 (8:09)
- **짧고 구체적인 핵심만** 담기
- 검증 가능한 규칙 위주

### 6. 실전 활용 사례 4가지

**사례 ① 자기 검증 규칙 (9:04) — 최고의 레버리지**
- "코드 작성 후 반드시 lint 실행하고 에러 0개 확인"
- "테스트 작성 후 `npm test` 통과 확인"
- Claude가 스스로 검증할 수 있는 규칙 = 실제로 지켜짐

**사례 ② 비즈니스 도메인 용어 정의 (9:49)**
- "유저 = 결제한 고객, 비회원 = 가입만 한 사람"
- 도메인 용어를 명확히 정의하면 코드 일관성 ↑

**사례 ③ 모듈식 관리와 @참조 활용 (10:52)**
- 프로젝트 루트: 전체 규칙
- 각 모듈 디렉토리: 모듈별 규칙
- `@파일명`으로 다른 문서 참조 가능

**사례 ④ 팀 공유 학습 시스템 (11:44)**
- CLAUDE.md를 Git으로 관리
- PR 리뷰에서 나온 피드백 → CLAUDE.md에 반영
- 팀 전체가 AI 코딩 품질을 공유/개선

### 7. CLAUDE.md 작성 3원칙 (13:14)

> 1. **짧게** — 200줄 이내, 핵심만
> 2. **구체적으로** — 검증 가능한 규칙

## 💡 핵심 인사이트

1. **CLAUDE.md는 "맥락"이지 "강제"가 아니다** — 검증 가능한 규칙을 넣어야 실효성 있음
2. **자기 검증 규칙이 최고의 레버리지** — "lint 통과 확인", "테스트 성공 확인" 같은 것
3. **200줄 초과 = 품질 하락** — 짧을수록 좋음
4. **도메인 용어 정의가 의외로 강력** — 비즈니스 로직 일관성 확보
5. **Git으로 관리하면 팀 공유 가능** — PR 피드백을 CLAUDE.md에 축적
